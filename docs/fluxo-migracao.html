6. Fluxo Ideal de Migracao

FASE 1: Validacao e Atualizacao de Dados

Objetivo: Garantir que nosso banco tem dados corretos e completos antes de gerar views.

  • 1.1 Comparar API /persons com person_fisica (CPFs, nomes, enderecos)
  • 1.2 Comparar API /contracts com servicos_historico (hash = contrato)
  • 1.3 Comparar API /items com itens do servicos_historico
  • 1.4 Comparar API /discounts com bolsas_descontos
  • 1.5 Puxar dados de 2026 que podem nao estar no banco (matriculas, contratos novos)

Quem faz: Script automatizado (Python/SQL) + validacao manual

FASE 2: Completar Views Financeiras

Objetivo: Reescrever/criar as views pendentes com dados validados.

  • 2.1 Reimportar SHABILITACAOFILIAL no TOTVS (adicionar turnos Manha/Tarde EI UN2)
  • 2.2 Reimportar SHABMODELOPGTO (depende de 2.1)
  • 2.3 Reescrever SCONTRATO — 1 contrato por aluno/periodo, ligado ao plano correto
  • 2.4 Criar SPARCELA — parcelas reais com valores, datas, servico, responsavel financeiro
  • 2.5 Validar/criar SBOLSAALUNO — bolsas aplicadas por aluno
  • 2.6 Criar SLAN e FDADOSPGTO se necessario

Quem faz: Guilherme + assistente de IA (SQL views)

FASE 3: Importacao e Validacao Final

Objetivo: Carregar no TOTVS e confirmar que tudo bateu.

  • 3.1 Importar views financeiras no TOTVS na ordem correta
  • 3.2 Validar totais: alunos por ano, contratos, valores
  • 3.3 Cross-check API Gennera vs TOTVS Oracle (amostragem)
  • 3.4 Documentar divergencias para ajuste manual

Quem faz: Guilherme + consultor TOTVS

7. O Que a API Resolve Que o Banco Nao Resolve

ProblemaSem APICom API
Dados de 2026 So temos rematriculas no banco. Faltam contratos e matriculas completas. API retorna matriculas, contratos e faturas de 2026 em tempo real.
Contrato detalhado servicos_historico so tem linhas de cobranca. Precisamos derivar contrato. /contracts/:id/detailed retorna compras (purchases), pessoa, dia de vencimento.
Faturas reais (SPARCELA) Precisamos calcular parcelas a partir de valores brutos do historico. /contracts/:id/invoices retorna cada fatura com valor, status (pago/pendente), data.
Bolsas por aluno So temos catalogo de bolsas (bolsas_descontos). Nao sabemos quem recebe qual. Contrato detalhado mostra descontos aplicados em cada compra do aluno.
Responsavel financeiro servicos_historico tem CPF do responsavel mas nao o CODCFO. API retorna idPerson do responsavel financeiro direto no contrato.
IDs inconsistentes enrollment.id_person e servicos_historico.id_pessoa sao namespaces diferentes. API usa o MESMO namespace que servicos_historico. Podemos criar tabela de mapeamento.
